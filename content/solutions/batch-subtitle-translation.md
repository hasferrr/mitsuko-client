## The real problem with translating a season one file at a time

Translating one subtitle file is easy enough. Translating a season, course, or playlist is where problems start.

The biggest issue is not speed. It is **terminology drift**: the same name, product term, or phrase changing from one file to another.

That is how episode 3 gets "Moon Key" while episode 7 gets "Lunar Key." A batch project with shared context keeps those choices stable across every file.

## Term drift, shown on a real line

Here is the same recurring reference handled two ways: isolated translation vs. a context-guided batch draft.

| Type | Line |
|---|---|
| Series batch sample | Episode 03: The Council calls the artifact the Moon Key again. |
| One-file translation risk | Dewan menyebut artefak itu Kunci Rembulan lagi. |
| Context-guided batch draft | Dewan menyebut artefak itu Kunci Bulan lagi. |

`Kunci Rembulan` and `Kunci Bulan` both mean "Moon Key" in Indonesian. `Rembulan` is poetic. `Bulan` is the everyday word.

Both are "correct," and that is the trap. An isolated translator picks whichever feels natural in that file.

A batch project with shared context uses the version the project already chose and keeps it across every file. Viewers never see the drift.

### The same pattern in course content

| Type | Line |
|---|---|
| Course subtitle sample | Drag the clip into the timeline, then export the captions. |
| Literal draft risk | Seret klip ke garis waktu, lalu ekspor keterangan. |
| Workflow-aware subtitle | Seret klip ke timeline, lalu ekspor caption-nya. |

`garis waktu` is the dictionary translation of "timeline." `keterangan` is the dictionary translation of "captions."

But in a video-editing course, viewers often expect the product terms `timeline` and `caption` to be kept or adapted the way the software labels them.

A batch project lets teams keep product and workflow terms consistent across every lesson instead of re-deciding them per file.

## How a batch project works

The design goal is simple: set it up once, review everything together.

1. **Upload multiple subtitles at once.** Drag in a season of SRT files, a mix of SRT/VTT/ASS, or a folder of course captions. They land in one queue.
2. **Apply shared language and instruction settings.** One target language, one model, one set of custom instructions, and one context document - applied to every file in the batch. You don't re-enter the glossary twelve times.
3. **Review and export translated files in bulk.** Files finish into the same project, so you can QC them as a set and export the whole batch when the season is ready.

## Why "one queue, one set of rules" beats parallel one-offs

You could run twelve separate translations in twelve tabs. People do.

The catch is that consistency then depends on a human remembering every term decision across all twelve files.

That is hard for people and easy for software, *if* the software is given one shared context to work from.

A batch project makes the shared context clear and enforced. The recurring-name decision lives in the project, not in someone's head.

Episode 12 inherits the same rule as episode 1. That is the moat: not just faster translation, but **consistent** translation that does not need a cleanup pass afterward.

## Who this is for

- Episodes, courses, playlists, and recurring client work where many files share the same language, context, and delivery rules.
- Teams that want one setup for many files instead of N independent setups.

## Formats

Batch projects accept **SRT, VTT, and ASS** in the same queue. You do not need a separate project per format.

Mix an `.srt` episode with an `.ass` special and they share the same context and settings.
