## Cat 2: Card-like divs → Card/CardContent (3 files)

Three files have identical divs that mirror Card+CardContent:

```
<div className="space-y-4 w-full p-4 mt-4 rounded-xl border border-input bg-card shadow-xs">
```

Convert to Card+CardContent, accepting Card's default ring and padding. Card provides `bg-card rounded-xl ring-1 ring-foreground/10`. The `border border-input` is replaced by Card's default ring. Keep `shadow-xs` and `mt-4 w-full`.

**Result:**
```tsx
<Card size="sm" className="mt-4 w-full shadow-xs">
  <CardContent className="space-y-4">
    ...
  </CardContent>
</Card>
```

Card size="sm" gives `py-3 gap-3` + CardContent `px-3`. The `space-y-4` on CardContent handles child spacing.

## Cat 9: Hardcoded colors → design tokens (2 definite + 1 borderline)

### Definite: `text-red-500` → `text-destructive`
- `manage-uploads-dialog.tsx` line 169: `className="text-red-500"` on a delete Button → `className="text-destructive"`

### Borderline: Multi-color status in sortable-batch-transcription-file.tsx
- Line 104: `text-red-600` for error status description
- Per critical rules, per-feature accent colors are intentional. The `text-red-600` is part of a multi-color system (`green/blue/red/yellow`). Only replacing red but not the others creates inconsistency. **Skip this** — it's a status color system, not a design-token candidate.

### Intentionally skip: Green "Mark" button
- `batch-extraction-view.tsx` line 403: `bg-green-500 hover:bg-green-600 text-white border-0`
- This is a per-feature accent (green for "Mark" action in extraction). Per critical rules, keep as-is.
